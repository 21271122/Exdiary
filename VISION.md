# Exdiary 项目愿景与设计原则

本文件记录项目的核心方向、设计原则和边界判断，用于指导后续所有功能决策。
这些结论来自对"实验记录 + AI 分析"这一方向的多轮讨论和反思。

---

## 一、项目定位

Exdiary 是一个**辅助研究者管理实验知识**的工具，不是一个**替代研究者做科研决策**的系统。

核心价值主张：**帮你节省整理和检索的时间，让你花更多时间在真正思考上。**

三个定位声明：

1. Exdiary 是**工具的延伸**——帮你归档、检索、可视化、做计算。这些是"我已知道我要什么，但手动做太慢"的事。
2. Exdiary 是**讨论的伙伴**——帮你发现异常、提出值得思考的问题、提醒你忽略的细节。但它不下结论，不做价值判断。
3. Exdiary **不是你的替代品**——不自动生成结论，不替你决定下一步方向，不替你写学术论证。

---

## 二、核心原则：AI 与人的边界

### 2.1 应该交给系统做的事（自动化）

这些工作本质上是**机械的、重复的、格式化的**，不产生新的科学判断：

- **文档整理**：自然语言 → 结构化实验记录。这是秘书工作，不是科研工作。
- **信息检索**：从大量实验中按条件查找。替代手动翻文件。
- **数据清洗**：从文本中提取数值（如"降解率 92%"→ 92）。替代手动录入 Excel。
- **可视化**：把已提取的数值画成图。替代 Origin/Excel 绘图操作。
- **统计计算**：相关性、趋势拟合、异常检测。替代手动算统计量。
- **格式化工**：生成实验方法段的论文草稿。替代从零开始排版。

### 2.2 不应该交给系统做的事（保留给人）

这些工作需要**判断力、品味、价值权衡、学术责任**：

- **自动生成结论**：结论是你的学术判断，不是文本生成任务。
- **替你决定下一步**：价值判断不能被外包。"值得研究什么"是科研的核心。
- **替你写讨论段**：论证你的实验为什么有意义——这是你作为研究者的核心产出。
- **不经确认的分析报告**：分析结果需要你逐条验证，因为你署名。

### 2.3 灰色地带：AI 辅助但人做最终判断

- **表征数据解读**：AI 可以提供参考意见（"这个 XRD 峰可能对应 TiO2 锐钛矿相"），但判断权在你（"这是仪器噪声还是真实信号？"）。
- **规律发现**：AI 可以提出假设（"你的数据中降解率随温度升高"），你来判断这个规律是真实的因果还是巧合。
- **异常检测**：AI 标注异常实验，但解释异常原因需要你的领域知识。

### 2.4 判断标准

当你在犹豫"这个功能该不该加"时，问自己：

> 这个功能是在**帮我省时间**，还是在**替我思考**？

前者值得做，后者需要慎重。边界可以逐步调整，但必须是有意识的调整，而非不知不觉的滑坡。

## 三、技术哲学：不预判结构，在分析时按需提取

### 3.1 核心洞察

实验记录有两个根本矛盾：

1. **实验类型极度多样**：光催化、XRD、电化学、水热合成……关心的数据各不相同。不存在一个万能的结构化 schema。
2. **同一实验类型内部高度分化**：同一个"钙钛矿太阳能电池"，不同课题组关心的数据不同。同一个人在不同阶段关心的数据也不同。
3. **科研本质是探索未知**：你最关心的变量，在做实验之前可能根本不知道它存在。预定义 schema 假设你提前知道什么重要——这本身是反科研的。

**结论**：不在记录时预判未来会怎么分析（**schema-on-write**），改为在分析时按需求从完整记录中提取所需数据（**schema-on-read**）。

### 3.2 记录端：保持通用和完整

- 当前 11 段通用结构保留。目的、材料、步骤、参数、观察、结果、结论、原始笔记——任何实验都能填。
- **不做领域专属模板**。不预设光催化模板、电化学模板等。
- 信息密度做在**原始笔记**里，而不是预定义的表单字段里。只要原始记录足够详细，任何时候想提取什么数据，都能从里面挖出来。

### 3.3 分析端：问题驱动，按需提取

每次分析时，系统针对用户的具体问题，从相关实验的完整记录中按需提取所需数据：

```
用户提问："对比我所有钙钛矿电池的 PCE 变化趋势"

系统行为：
  1. 语义搜索 → 找到相关实验（标签 + 标题 + 目的匹配）
  2. LLM 定向提取 → 从每个实验的 results/conclusion/notes 中提取 PCE 数值
  3. 展示提取结果 → 让用户确认/修正（关键步骤，不可跳过）
  4. 确定性计算 → 相关性、趋势、异常（用 Python 算，不用 LLM 算）
  5. 可视化 → 散点图、趋势线
  6. 发现提示 → 指出值得关注的模式和问题，而非给出结论
```

今天问 PCE 就提取 PCE，明天问稳定性就提取稳定性。不需要在写实验记录时预判未来会问什么。

### 3.4 提取结果缓存

```
第一次分析"PCE 趋势" → LLM 提取 15 条 PCE → 用户确认/修正 → 写入缓存
第二次分析"PCE vs 退火温度" → 读缓存中已确认的 PCE → 只需提取新维度"退火温度" → 合并分析
```

随着使用，缓存中积累的已验证数据越来越多。后续分析的提取成本越来越低，准确度越来越高。

### 3.5 分析的三个层次

| 层次 | 适用条件 | 分析方法 |
|---|---|---|
| **同类实验定量分析** | 用户选择了共享某种变量的实验，系统按需提取了可比较的数值指标 | 确定性计算（相关性、趋势、异常）+ LLM 语义解读 |
| **异类实验语义分析** | 用户选择不同类型的实验（如光催化+XRD+电化学） | LLM 做方法论交叉、知识链完整性检查，不做数值比较 |
| **全量元分析** | 所有实验（不分类别） | 统计 + LLM：成功率趋势、失败模式语义聚类、标签演化、知识复用追踪 |

三层各司其职，不强行把异类实验塞进同一数值空间。

## 四、交互设计原则

### 4.1 系统输出应该是"发现 + 提问"，而非"结论"

```
错误示范（结论式）：
  "你的实验表明煅烧温度升高有利于降解率提升，
   建议下一步测试 500°C 条件。"

正确示范（发现 + 提问式）：
  **数据呈现**（事实）：
  "选中 5 个实验。煅烧温度：300/350/400/450/500°C，
   降解率：78%/84%/89%/92%/91%。r=0.94。"

  **发现提示**（可能的方向，带不确定性标注）：
  "系统注意到：
   - 400-500°C 区间降解率趋于平台（从 89% 到 92% 到 91%），
     增益可能已接近上限
   - EXP-005（500°C）的降解率略低于 EXP-004（450°C），
     差异可能来自【测量误差 / 过高温度导致催化剂烧结 / 其他】
   - 你尚未在 350-400°C 之间做更细的梯度"

  **向你提问**（激发思考）：
  "建议思考：
   - EXP-005 降解率下降是真实的吗？需要重复吗？
   - 450°C 是否已经是该体系的最优条件？
   - 这些实验的降解率是用同一批 MB 溶液测的吗？"
```

### 4.2 每次 LLM 提取的数据必须经过人工确认

这不是信任问题，是责任问题。你署名，你确认。系统不能替你签这个字。

确认交互的设计原则：
- 提取结果高亮展示，不确定的数值标黄
- 用户可以修正、删除、补充
- 确认后的数据进入缓存，下次不再重复提取
- 这些标注行为本身也是有价值的——它们记录了你的判断过程

### 4.3 分析能力由提问驱动，不由字段预定义

分析页面不需要预定义"你可以分析什么"。但可以提供智能引导：

- "检测到 5 个光催化实验，常见比较维度：负载量、降解率、光源功率"
- "检测到 3 个 XRD 实验，常见比较维度：物相组成、结晶度、晶粒尺寸"
- "EXP-003 和 EXP-005 使用了不同的前驱体，建议比较产物差异"

这些引导是 LLM 快速扫描实验文本后的建议，不是预置模板。它们给用户灵感和起点，但不限制用户提问的方向。

## 五、功能边界清单

### 要做的事

- [ ] 按需数据提取器（分析时从完整记录中定向提取，动态生成 schema）
- [ ] 提取结果缓存 + 人工确认界面
- [ ] 分析管道分层：确定性计算（统计/可视化）→ LLM 语义解读
- [ ] 分析结果的"发现 + 提问"式展示
- [ ] 全量元分析看板（失败模式聚类、标签演化、复现性统计）
- [ ] 实验关系网络（样品关联、方法引用、知识链完整性）
- [ ] 数值型查询 API（从文本中提取后的缓存数据支持精确查询）
- [ ] 分析问题智能引导（扫描选中实验，提示可能的分析角度）

### 不做的事

- 自动生成实验结论
- 自动生成"下一步建议"（可以提示你已有的 next_steps 是否被执行，但不能替你决定方向）
- 自动生成论文讨论段
- 预设领域专属 schema（光催化模板、电化学模板等）
- 不经确认的数据分析报告

### 需要持续讨论的事

- 表征数据（XRD 谱图、CV 曲线等原始仪器数据）的接入方式：是保存原始文件本身，还是解析数据点，还是只记录结果结论？三种方式各有利弊，需要根据实际使用场景决定。
- 多人协作模式：当多个研究者共享一个实验库时，边界如何划分——各自的实验独立存储但可被搜索？还是有一个共享的全局视图？
- 外部文献关联：实验记录中引用的文献，是手动输入 DOI，还是做轻量级关联（存一个引用字符串）？

## 六、开发优先级

### 第一阶段：打通按需分析链路

**目标**：验证"schema-on-read"模式的核心流程。

1. 分析页面改造：从"选实验 + 输入文本问题"升级为"选实验 + 系统智能引导 + 按需提取 + 确认 + 可视化"
2. 按需数据提取器：给定实验列表和提取需求描述，LLM 动态提取结构化数据
3. 提取结果确认界面：展示、修正、缓存
4. 确定性计算集成：相关性、趋势拟合、异常检测（纯 Python，不调 LLM）

### 第二阶段：完善分析体验

**目标**：覆盖更多分析场景，提升交互质量。

5. 全量元分析看板（失败模式语义聚类、标签演化、复现性统计）
6. 实验关系网络可视化
7. 分析结果的"发现 + 提问"式展示
8. 数值型查询 API（基于缓存的数据做精确查询）

### 第三阶段：数据层强化

**目标**：支撑更大数据量和更复杂的查询需求。

9. SQLite 索引层（YAML 仍为数据源，SQLite 用于查询加速）
10. 分析模板（可复用的分析流程定义）
11. 更多分析引导场景

## 七、附：我们走过的思考历程

这些原则不是一开始就有的。以下是关键转折点：

1. **第一轮讨论**：提出了 8 大改进方向（可视化、文献、样品、DOE、设备、库存、论文导出、协作）。方向太散，没有聚焦，且隐含"添加更多结构化字段"的假设。

2. **第二轮讨论**：聚焦到"实验记录与管理 + 大数据分析"，提出了"通用 + 专属双层 schema"和"分析管道分层"。但被质疑：同一实验类型内差异大，预设领域模板太死板。

3. **第三轮讨论**：放弃"预设结构"思路，转向"schema-on-read"——不在记录时预判分析需求，在分析时按需提取。这是核心方向性转折。

4. **第四轮讨论**：质疑 AI 在科研中的角色边界——AI 做太多会不会让科研主体从人变成机器？确立了"AI 提问而非 AI 下结论"的交互原则，以及"帮你省时间 vs 替你思考"的判断标准。

这些讨论的痕迹保留在此，是为了让未来的改动都有据可查：当你犹豫"这个功能合不合适"时，回来读第三节和第五节。
